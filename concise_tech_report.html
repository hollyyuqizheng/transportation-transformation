    • New York MTA Subway Back to top
      1. Source of original data:
        New York City MTA turnstile usage
        New York City MTA station information
        The source is reputable, since the data comes from the official New York public transit website.
        We managed to gather monthly cumulative data for 250 subway stations in New York City.
      2. Additional considerations while collecting data?
        The subway ridership data contains the number of entry and exit at each subway turnstile over 4-hour period on a given day.
        The dataset is publicly available from the New York MTA site. In addition, becaue the original dataset only contains the number of turnstile swipes,
        the information would not contain identifying information of those that take the New York subway.

      3. How is the data distributed?
        The ridership dataset contains data for all MTA stations, and the usage at each station is measured every 4 hours,
        so the dataset should have a comprehensive coverage of all subway stations in the city.

      4. Are there any duplicates? Missing values?
        We assume that there are not duplicates from the original data from MTA. Each entry is marked by the id of the subway station.

      5. Are there any data type issues? Did you have to throw away any data?
        We had issue when matching each turnstile data entry with the location coordinate of the subway station,
        because the naming convention in the turnstile dataset is different from the convention in the station info dataset.
        For example, a station might be called "Grand Central - 42 St" in one dataset and "Grd Ctrl" in the other one.
        We managed to fix the above issue by manually editing the station names in one dataset so that the name matches its counterpart in the other dataset.
        For example, we replaced ""Times Square"" with ""Times Sq"" so that both datasets have the same naming convention for this location.
        Before this fixing, only around 60% of station names match. After the fix, we managed to match around 70% of all the turnstile stations with their coordinates.
        For the rest of the stations whose naming conventions across the two original data sources don't match, we decided to ignore those entries from the turnstile data.

    • Chicago CTA subway "L line" Back to top
      1. How did you collect your data? How is the source?
        The Chicago Transit Authority has publicly available open data sets regarding ridership on its L line train stations
        and we were able to access data sets containing information on average monthly train ridership per station.
        This dataset tallies ridership information monthly from 2001 through 2019.
        The source is reputable; the information comes directly from the Chicago Transit Authority website.

      2. Additional considerations while collecting data?
        While subway riders are likely not aware of the extent to which this data is collected, it is reasonable in metropolitan cities to assume that
        ridership numbers are monitored for the purpose of improving transportation services and deciding on locations for new development.

      3. How is the data distributed?
        There are 1717 entries in this dataset, with each entry representing one Chicago subway station's usage frequency for a month.
        The dataset covers 143 different subway stations.
        There is variation in ridership between train stations but overall the data seems to be generally uniform in how ridership is distributed at each station monthly based on sampling.

      4. Are there any duplicates? Missing values?
        There is no duplicate station information. There are no missing values. Each row in the dataset is completely filled.

      5. Are there any data type issues? Did you have to throw away any data?
        There do not seem to be any data type issues. All data is consistent with the data type denoted in the column.
        One entry from the original ridership dataset contains information about a station in Chicago that does not have a matching station id in the station information set.
        We decided to omit this station's entry, since this station appears to only exist in the dataset once for the year 2019.

    • Chicago Taxi Back to top
      1. How did you collect your data? How is the source?
        The City of Chicago's data portal has the taxi data publicly available -- we used the Sodapy API to scrape the data.
        Original source
        The source is reputable. The original data comes from the City of Chicago's offical government data portal.

      2. Additional considerations while collecting data?
        I don't think that the taxi riders are aware that the information about their data is being collected, but the taxi companies have partnered with the City of Chicago,
        so the drivers know that the ride time/location data is being collected.

      3. How is the data distributed?
        There are 2 million rows (one for each ride) and 23 columns (different descriptors of the ride). We mainly care about the datetime and the lat/lon of pick-up and dropoff locations.
        We want to find daily ridership in different parts of the city. We have more than 20k rows per day, so I think there's enough information.
        The data is not uniform since there are more taxi rides in areas we expect, such as O'Hare International Airport and the Loop (major business and tourism center).
        There is a max of 16,513 taxi pickups from the Near North Side (Magnificent Mile and Gold Coast areas), and a min of 1 taxi pickup from the Mount Greenwood neighborhood (far Southwest side).
        A CSV histogram of pickups by community area is in data/chicago/pickups_histogram.csv.

      4. Are there any duplicates? Missing values?
        There are no completely duplicated rows. There might be partially duplicated information, but it's difficult to tell.
        Yes, there are missing values in almost every column. In particular, there are missing values in the pick-up and drop-off lat/lon columns, which is crucial for our analysis.

      5. Are there any data type issues? Did you have to throw away any data?
        The datetime format is in "Floating Timestamp" format, which I subscript in order to find the date (for the daily ridership data). The lat/lon data is in float format, which we can crosscheck with OpenStreetMap or other mapping APIs.
        We're throwing away data where the lat/lon pickup and drop-off information isn't available. We're going to throw away columns that aren't relevant to our goals.
